## Outcome
Six MCP tools split across two conceptual families. **Localize** family: `mask_set`, `mask_set_shape`, `mask_apply` produce an alpha mask on a clip and bind downstream effects to it. **Remove background** family: `effect_chroma_key`, `effect_chroma_key_advanced`, `effect_object_mask` are typed wrappers over `effect_add`. Coordinates are normalized `[0, 1]`. Multi-mask stacks via multiple calls. Static shapes only in v1.

## Intent
**Trade-off hierarchy:**
1. Semantic clarity (split localize vs remove-BG) over API count
2. Resolution-independence (normalized coords) over pixel precision
3. Composition over new primitives — reuse patcher + catalog
4. Defer keyframed masks to Spec 7 extension

**Preferences:**
- Prefer normalized `[0, 1]` coord space
- Prefer static shapes in v1; animation layered on later
- Prefer reusing existing `insert_effect_xml` / `reorder_effects` from Spec 1-2
- Prefer Pydantic for typed param validation

**Escalation triggers:**
- If Kdenlive's rotoscoping filter uses a fundamentally different alpha-routing mechanism than "stack order + alpha_operation property" — stop, inspect a hand-authored Kdenlive project with masks, report.
- If basic chroma key's MLT service string cannot be determined from the catalog — stop and ask which service to use.
- If `object_mask` requires an external model (AI subject detector) that isn't part of Kdenlive's stock install — stop and report.

## Requirements

1. `mask_set(clip_ref, type, params)` inserts a mask filter (rotoscoping | object_mask | image_alpha) at the top of the clip's stack. Returns the new filter's `effect_index`.
2. `mask_set_shape(clip_ref, shape, bounds|points, feather, alpha_operation)` converts rect/ellipse/polygon to a rotoscoping filter via spline sampling; delegates to `mask_set`.
3. `mask_apply(clip_ref, mask_effect_index, target_effect_index)` ensures the target filter is stack-below the mask filter (reorders if needed) so it consumes the mask's alpha.
4. `effect_chroma_key(clip_ref, color, tolerance, blend)` inserts a basic chroma key filter at top of stack.
5. `effect_chroma_key_advanced(clip_ref, color, tolerance_near, tolerance_far, edge_smooth, spill_suppression)` inserts an advanced chroma key filter.
6. `effect_object_mask(clip_ref, enabled, threshold)` inserts Kdenlive's object mask filter.
7. All coords are normalized `[0, 1]` — strict range validation.
8. Ellipse shape samples at 32 points by default (configurable).
9. Polygon requires ≥ 3 points.
10. Chroma key color accepts `#RRGGBB` / `#RRGGBBAA` / `int`; emitted as MLT `0xRRGGBBAA`.
11. All write-mutating tools auto-snapshot and return `snapshot_id`.
12. Full test suite passes with zero regressions.

## Edge Cases

- **Polygon with duplicate consecutive points** — pass through; Kdenlive handles it.
- **Ellipse with sample_count < 4** — error (degenerate polygon).
- **mask_apply on clip with exactly 2 filters where mask is the second** — reorder so mask is first; works correctly.
- **Multiple masks on one clip** — each is a separate `mask_set*` call. `mask_apply` pairs each mask with its target(s). Caller responsibility to order correctly.
- **Chroma key color with no alpha channel** (`#00FF00`) — default alpha `ff` appended → `0x00ff00ff`.
- **Object mask when not in catalog** (not all Kdenlive builds have it) — `_err` with pointer to effect_list_common for availability check.
- **mask_set with `type="image_alpha"`** but `image_alpha` not yet implemented — `_err("image_alpha mask type not yet implemented — use type='rotoscoping' or 'object_mask'")`. (Reserved for Spec 7+.)
- **Bounds with w=0 or h=0** — error, listed as invalid.
- **Feather larger than clip resolution** — pass through; Kdenlive clamps at render time.
- **mask_apply where target_effect_index is itself a mask filter** — error; cannot mask a mask.

## Out of Scope

- Animated mask shapes (keyframed splines) — Spec 7
- `image_alpha` mask type (use an image's alpha as mask) — deferred
- Auto-generated complex shapes (star, heart, text) — Spec 7+
- Mask hierarchy / mask groups
- UI tools (mask drawing)
- Per-point feather (currently global feather)
- Keyframed feather / alpha_operation changes
- Advanced spline types (bezier control handles) — v1 uses linear connections between points

## Constraints

### Musts
- All acceptance criteria.
- Python 3.12+, Pydantic v2.
- Normalized `[0, 1]` coordinate space everywhere.
- Use `patcher.insert_effect_xml` / `reorder_effects` — no new mutation primitives.
- Ellipse default sample_count = 32.

### Must-Nots
- Must NOT modify patcher or existing pipelines.
- Must NOT accept pixel coordinates (only normalized).
- Must NOT auto-order multiple masks on one clip — caller specifies order by call sequence.
- Must NOT silently clamp out-of-range coordinates — reject with error.

### Preferences
- Prefer typed enum Literals for `alpha_operation`, `shape`, `type`.
- Prefer explicit errors over silent default substitution.
- Prefer regex-scoped XML attribute writes (reuse Spec 2 pattern where applicable).

### Escalation Triggers
- If rotoscoping MLT property names differ from the XML structure (`shape`, `spline_is_open`, `feather`, `alpha_operation`), stop and confirm via a hand-authored fixture.
- If alpha routing requires more than stack-order + one property on the target filter, stop and ask.
- If `chroma_key` and `chroma_key_advanced` MLT service names are not unambiguous in the catalog, stop and ask which to use.
